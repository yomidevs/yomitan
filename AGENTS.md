# AGENTS.md

## Project Summary

Manabitan is a browser extension fork of Yomitan focused on faster dictionary imports, smaller storage use, and faster lookups. Most product code lives in `ext/`, build and code generation logic lives in `dev/`, and automated tests live in `test/`.

This repository still contains many historic `yomitan` names in package metadata, scripts, test paths, and build outputs. Treat those as intentional unless the task is explicitly about renaming them.

## Environment

- Use Node.js `>=22.0.0`.
- Install dependencies with `npm ci`.
- The repository uses ESM (`"type": "module"` in `package.json`).

## Repository Map

- `ext/`: extension source and packaged assets.
- `ext/js/background/`: service worker and backend logic.
- `ext/js/app/`, `ext/js/display/`, `ext/js/pages/`: popup, search, settings, page controllers, and other UI code.
- `ext/js/dictionary/`: dictionary import, storage, OPFS, sqlite, and worker code.
- `ext/js/language/`: language-specific transforms and processors.
- `ext/data/`, `ext/css/`, `ext/images/`, `ext/fonts/`: shipped assets and schemas.
- `dev/`: build scripts, manifest tooling, validators, vendored libraries, and codegen inputs.
- `test/`: Vitest suites, fixtures, Playwright tests, and browser-specific end-to-end tests.
- `types/`: `.d.ts` coverage for runtime, dev tooling, and tests.
- `docs/`: contributor documentation, dictionaries, templates, and development notes.
- `benches/`: benchmark suites.

## Working Rules

- Make targeted changes. Avoid broad refactors or style churn unless the task requires it.
- Follow the existing plain-JavaScript style and module layout.
- Preserve user-facing `Manabitan` branding where it already exists.
- Do not mass-rename legacy `yomitan` identifiers, paths, or outputs unless the change is explicitly requested.
- Prefer modifying source files, not generated artifacts.

## Build and Test Commands

- `npm run build`: build extension packages for the standard targets.
- `npm run test:build`: dry-run the build without creating files.
- `npm test`: full validation suite.
- `npm run test:fast`: faster common validation path.
- `npm run test:js`: ESLint.
- `npm run test:ts`: TypeScript validation across main, dev, test, and bench code.
- `npm run test:css`: CSS linting.
- `npm run test:html`: HTML validation.
- `npm run test:json`: JSON format and type validation.
- `npm run test:md`: Prettier check.
- `npm run test:unit`: Vitest unit tests.
- `npm run test:unit:options`: options upgrade tests.
- `npm run test:playwright`: Playwright suite.
- `npm run test:playwright:integration`: Chromium integration Playwright run.
- `npm run bench`: benchmarks.
- `npm run build:libs`: rebuild third-party library bundles.
- `npm run license-report:html`: regenerate `ext/legal-npm.html`.
- `npm run anki:css-json:write`: regenerate Anki structured-content CSS JSON when related styling changes.

## Generated and Special Files

- `ext/manifest.json` is generated from `dev/data/manifest-variants.json`. Do not hand-edit it.
- `ext/legal-npm.html` is generated by `npm run license-report:html`.
- `builds/` contains generated release artifacts.
- `npm run test:unit:write` rewrites expected output data for selected tests. Review those diffs carefully before committing.

## Testing Guidance

- For language-transform changes, run the relevant tests in `test/language/` plus `npm run test:unit`.
- For dictionary storage, import, worker, or backend changes, run `npm run test:unit` and `npm run test:build`. Add browser or integration coverage when the change affects actual extension behavior.
- For UI, HTML, or CSS changes, run `npm run test:html`, `npm run test:css`, and the most relevant unit or Playwright tests.
- For markdown-only documentation changes, at minimum run `npx prettier AGENTS.md --check`.

## Local Development Notes

- Chromium development can usually load `ext/` directly as an unpacked extension.
- Backend or manifest changes may require reloading the extension, and manifest changes may require rerunning `npm run build`.
- Playwright requires additional setup, including installed browsers and external dictionary test data from the `dictionaries` branch described in `CONTRIBUTING.md`.
- Browser end-to-end scripts have extra prerequisites such as `geckodriver`, `selenium-webdriver`, `adb`, or built extension packages. Do not assume those tools are available.

- Run the narrowest relevant validation for the files you changed.

- If you touch generated files or rewritten fixtures, mention why in your summary.
